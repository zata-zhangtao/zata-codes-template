# Migration（行为保持型迁移）

## 概述

任何目标为「替换实现、保持行为」的改动都用这个 skill：框架迁移、大版本升级、运行时/库替换、语言移植，或大型机械式重构。

AI 代理在大型迁移上产生海量错误，几乎从来不是模型的问题，而是两个方法缺陷：

1. **一次性大改（big-bang）**：把整个项目一把梭重写。diff 大到不可审、错误隐藏，而且一旦工作量超出单次上下文，代理就会失去掌控。
2. **没有 oracle**：动手前没有捕获任何可执行的「正常」定义，于是「功能保持」永远无法被证明，只能靠赌。

这个 skill 消除这两点。一次迁移的成败，在第一次编辑之前就由「是否存在行为基线」决定了。之后的一切只是针对该基线做小步、可验证、可回滚的推进。

> **核心原则：** 先让「功能」变成可执行的验证。然后小步迁移，每一片都对基线验证后再进入下一片。当前片没对基线变绿之前，绝不开始下一片。

## 何时启用

当用户想做以下事情时启用：

- 在框架之间迁移（如 Flask→FastAPI、Vue 2→Vue 3、Express→Nest、Webpack→Vite）
- 升级一个带破坏性变更的大版本
- 替换运行时、ORM、HTTP 客户端、状态库、构建工具或其他基础依赖
- 把一个模块或服务移植到另一种语言或范式
- 做大型仓库级机械重构，且用户明确要求「保持功能」/「keep it working」

**不要**用这个 skill 去新增或修改功能。迁移是纯粹的行为保持型变换。如果行为必须改变，那是另一项任务、放进另一个提交，在迁移之前或之后做——绝不夹带进某个迁移片里。

## 不可协商的规则

- **迁移 ≠ 功能 ≠ 顺手重构。** 一个提交只做一件事。迁移片里不许夹带「反正都在改了」式的改动。
- **基线先于编辑。** 在行为基线于*旧*代码上变绿并提交之前，不迁移任何生产代码。
- **上一片没对基线变绿，下一片不许开始。** 这是硬门禁，不是建议。
- **只允许小而可回滚的 diff。** 如果一片的 diff 大到无法一次审完，它就不是一片——拆开。
- **不要相信模型记忆里的目标 API。** 始终对照目标的真实文档或一份已知可用的范例来迁移，并锁定版本。幻觉 API 是仅次于 big-bang 的第二大迁移错误来源。
- **不许静默跳过。** 如果某片无法对基线验证，明确暴露出来。绝不把它标成「已迁移」。

## 工作流

### 阶段 0：把迁移定义为契约

用平实语言说清三件事并记录下来（记录位置见阶段 1）：

- **From → To：** 精确的源与目标，带锁定的版本号。
- **不变量（invariant）：** 哪些可观测行为**必须**保持完全一致（这将成为阶段 2 的基线）。
- **允许改变的部分：** 内部实现、文件布局、写法——一切用户观察不到的东西。
- **非目标：** 明确写上「迁移期间不新增功能、不改行为、不顺手重构」。

如果迁移目标本身含糊或自相矛盾，先运行 `requirement-sanity-reviewer` skill。

### 阶段 1：发现项目（不要假设）

这个 skill 与具体项目无关（project-agnostic）。运行时去发现真实信息，而不是硬编码：

- **命令：** 找出项目真实的 test、lint、type-check、build、run 命令（查看 `justfile`/`Makefile`、`package.json` scripts、`pyproject.toml`、CI 配置）。记下确切命令——每一片都要跑它们。
- **真实入口：** 识别对*用户*而言「功能」由什么定义——HTTP 路由、CLI 命令、UI 流程、worker job、对外发布的库 API、数据迁移。基线必须落在这些地方。
- **影响面：** 枚举迁移触及的每一个文件 / 调用点 / 模块。用仓库级搜索（`rg`）找出对旧框架/旧 API 的引用。这份清单驱动切片——把它落盘，别只放在上下文里。
- **工作区：** 决定持久状态放哪（用 `planning-with-files` 承载切片表、基线清单和进度日志；用 `prd` 承载迁移计划 + 验收清单）。

不要问仓库自己能回答的问题。

### 阶段 2：冻结行为基线 —— oracle（最关键）

在**最高可观测层**捕获行为，因为内部实现正是要变的东西。绑在旧框架内部的单元测试*不是*有效的 oracle——它们会随代码一起被重写。把行为锁在用户真正观测到的边界：

| 面（Surface） | oracle 手法 |
|---|---|
| HTTP API | 对 请求 → 响应 录 golden/characterization 快照（状态码、关键 header、body），覆盖代表性 + 边界 + 错误用例 |
| CLI | 录 argv → stdout/stderr/退出码/文件副作用 |
| UI / 用户流程 | 用 Playwright 覆盖用户可见结果（用 `playwright-e2e` skill）；只在 UI 稳定处用视觉快照 |
| 库 / 公共 API | 对公共签名与返回值做 characterization 测试 |
| 数据 / schema | round-trip + 样本数据 in → out 校验；migration up/down |

规则：

- 基线必须在**旧代码上变绿**之后再开始迁移，并提交。它就是契约。
- 对确实无法自动测试的行为，写一份**人工验证脚本**：记录步骤 + 预期观察，最后由人执行。
- 记录一份**基线清单（Baseline Manifest）**（见 [references/migration-checklist.md](references/migration-checklist.md)）：锁了哪些行为、在什么层级、各自的运行命令。

### 阶段 3：切片（Strangler）

把迁移拆成一个个可独立验证、可独立回滚的增量。

- 按 模块 / 路由 / 组件 / 调用点 切——以「每个 diff 小到可审」为准。
- 优先**新旧并存**：在新路径搭建期间保持旧路径可运行（适配器、shim、并行路由、feature flag），让每一片都能单独上线或回滚。
- 排序用 叶子优先（依赖最少的先迁）或 高风险优先——选一种并记下理由。
- 把计划外置成一张**切片跟踪表（Slice Tracking Table）**（形状见 references 文件）：`切片 → 状态（pending/in-progress/migrated/verified）→ 是否已对基线验证 → 回滚提交`。用 `planning-with-files` 落盘，别放上下文里。

如果用户要求「一次性全迁」，拒绝并改为切片——说明 big-bang 正是他们想避免的错误的根因。

### 阶段 4：逐片循环（安全网）

对每一片，按顺序重复：

1. **收紧任务范围。** 交给执行者：这一片的文件、契约（不许变的行为）、目标框架的*正确写法*（一段真实范例或文档摘录来照抄——不靠记忆）、以及必须仍然通过的基线测试。
2. **做最小的机械变换。** 只做迁移——不重构、不清理、不做迁移并不强制的改名。
3. **跑验证回路。** 对受影响范围跑 发现到的 test + lint + type-check 命令，然后跑受影响面的**基线 oracle**，并**对旧 vs 新行为做 diff —— 必须完全一致**。
4. **绿了：** 提交这一片（小、可回滚）。更新切片跟踪表。可选：对该片 diff 跑 `code-reviewer`。
5. **红了：** 停。不许前进。只修或只回滚*这一片*——很便宜，因为 diff 小、且基线是仲裁者。记录失败与解决过程（三振协议：定向修复 → 换一种方法 → 重新审视假设/上报）。

**硬门禁：** 当第 N 片对基线还是红的时候，绝不开始第 N+1 片。

### 阶段 5：最终收尾

当所有片都迁完：

- 对完整迁移后的代码跑**整套基线**——全绿。
- **仓库级搜索断言：** 证明不再有对旧框架/旧 API 的引用，且预期的新引用已存在（两者都用 `rg`）。
- **拆掉脚手架：** 删除 shim、适配器、双路径，以及被 strangler 替换掉的旧代码。
- **完整验证：** 跑项目发现到的完整 test/lint/build、e2e，并通过真实入口至少跑一遍真实应用。
- **对账计划：** 完成迁移 PRD 的验收清单，记录确切的验证命令与结果。

## 规模化：编排（仅在显式 opt-in 时）

当迁移有大量同构改动点（例如成百上千个相同的调用点变换），这正是典型的多 agent 迁移形态：**发现改动点 → 每个点在隔离 worktree 里变换 → 各自对基线验证 → 对账**。只有当用户**显式选择**运行 workflow / 多 agent 时才用编排——不要自动触发。上面那套「先基线、再验证」的纪律仍然约束每一个被变换的点。

## 护栏

- 迁移、重构、功能各自独立提交——绝不混在一起。
- 锁定源与目标的版本号；对照目标真实文档或已知可用范例来迁移，绝不靠模型记忆。
- 把状态外置——长迁移会超出单次上下文。切片表、基线清单、进度日志必须落盘。
- 在新路径被证明之前保持旧路径可运行，让回滚永远只差一个提交。
- 把过大的切片 diff 当作缺陷：一直拆到每个 diff 都可审。
- 绝不为了让某片通过而降低基线断言。如果行为确实必须改变，停下来上报用户。

## 验证

项目自己发现到的命令就是门禁——跑它们，别自创。每一片、以及收尾时，至少：

- 对受影响范围跑 发现到的定向测试命令，收尾时再跑全量
- 跑发现到的 lint 与 type-check 命令
- 跑捕获好的、受影响面的基线 oracle（行为 diff 必须为空）
- 在最终收尾时跑 e2e/UI 命令并真实运行一次应用

把确切命令与通过/失败结果记进进度日志和 PRD 验收清单。

## 与其他 skill 的关系

- `requirement-sanity-reviewer`：当迁移目标本身不清晰或自相矛盾时，先跑它。
- `prd`：创建迁移计划及其验收清单（交付门禁）。
- `planning-with-files`：在长周期迁移里把切片表、基线清单、进度日志、错误日志落盘。
- `code-reviewer`：评审每一片的 diff 以及最终收尾的 diff。
- `playwright-e2e`：为 UI 迁移搭建用户流程基线。
- `deep-research`：当你需要目标框架当前、特定版本的迁移指引（来自外部资料）时。

## 示例

- 「把后端从 Flask 迁到 FastAPI，迁移后功能要完全一致。」
- 「把这个项目从 Vue 2 升级到 Vue 3，不改变任何用户可见行为。」
- 「在整个代码库里替换掉我们的 HTTP 客户端库，保持每个端点行为不变。」
- 「把这个模块移植成 TypeScript；调用方看到的一切都不能变。」
- 「我们要把构建从 Webpack 迁到 Vite —— 保持应用正常工作。」
